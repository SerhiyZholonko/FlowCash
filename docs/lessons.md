## 2026-06-27 — Навігація з головного екрана

**Контекст:** картки на `HomeView` вели не туди, куди вказує їх назва.

- Картка **"БАЛАНС"** додавала route `.budgets` → відкривалися "Бюджети" замість рахунків.
  Виправлено: додано `case accounts` в `HomeRoute`, обробку в `.navigationDestination`
  (`AccountsView()`), кнопка тепер `path.append(.accounts)`.
- Картка **"ДОХІД"** вела на `StatsView`, який показує **лише витрати**.
  За рішенням користувача — тепер відкриває `TransactionsView`, відфільтрований на доходи.
  Реалізація: `TransactionsViewModel` отримав `typeFilter: TransactionType?` (+ init,
  `filtered`, `navigationTitle`); `TransactionsView` — `init(typeFilter:)`; новий
  `HomeRoute.income` → `TransactionsView(typeFilter: .income)`.

**Висновок:** перевіряти відповідність назви UI-елемента його дії/призначенню екрана.
Назва картки ("БАЛАНС", "ДОХІД") має вести на змістовно відповідний екран.

## 2026-06-27 — Накладання календаря в AddTransactionView

**Контекст:** при виборі дати графічний `DatePicker` показувався як `.overlay` на `chipsRow`
зі зсувом `.offset(y: 220)` і `.zIndex(10)`, та накладався на `categoryGrid`.

**Причина:** `.overlay` належить `chipsRow`, який у `VStack` стоїть **перед** `categoryGrid`.
Дочірні елементи `VStack` малюються в порядку оголошення, тож `categoryGrid` малюється поверх.
`.zIndex` діє лише серед siblings у межах одного контейнера — між дочірніми `VStack` не допомагає.

**Виправлення:** прибрано overlay-хак, календар винесено в `.sheet` (стиль як у `noteSheet`).

**Висновок:** уникати overlay із магічним `offset` між елементами `VStack`. Для модальних
елементів використовувати `.sheet`/коренева `ZStack` overlay. `.zIndex` не керує порядком
між дочірніми елементами `VStack`.

## 2026-06-27 — Клік по всьому рядку "Найбільші витрати"

**Контекст:** рядки реагували на тап лише по непрозорих елементах (іконка, текст, прогрес),
порожній фон/відступи не клікались.

**Причина:** `Button` (і взагалі hit-testing) у SwiftUI реагує лише на непрозорі ділянки вмісту;
прозорі відступи без фону не є hit-test областю.

**Виправлення:** додано `.contentShape(Rectangle())` до `topExpenseRow` після відступів.

**Висновок:** для клікабельності всієї площі рядка з прозорими зонами — `.contentShape(Rectangle())`.

## 2026-06-27 — Накладання великого заголовка на пошук (TransactionsView)

**Контекст:** на екрані "Доходи" (`TransactionsView`) великий заголовок накладався на поле пошуку.

**Причина:** `.searchable(placement: .navigationBarDrawer(displayMode: .always))` змушує поле
пошуку показуватись завжди й одночасно з великим заголовком — на iOS 26 вони накладаються.
Також у тулбарі була декоративна `Image("magnifyingglass")` (не кнопка), що дублювала пошук.

**Виправлення:** placement змінено на `.automatic` (поле пошуку ховається до потягування вниз,
на старті — лише заголовок, як на "Статистиці"); прибрано декоративну іконку лупи.

**Висновок:** для великого заголовка не форсувати `.always` у `.searchable` — використовувати
`.automatic`. Декоративні `Image` у тулбарі плутають (виглядають клікабельними) — уникати.

## 2026-06-27 — Заставка завантаження

**Контекст:** при старті дані з'являлися не одразу — користувач бачив порожній стан.

**Реалізація:**
- `HomeViewModel.isLoading = true` на старті; `defer { isLoading = false }` у `loadData()`
  скидає прапорець після першого завантаження навіть при помилці; стає false лише раз
  (подальші `loadData()` заставку не показують).
- `SplashView` (`Core/Launch/`): лого пульсує + обертова дуга-спінер.
- Лого — окремий `LaunchLogo.imageset` (копія іконки), бо `AppIcon` набір недоступний як
  звичайний `Image` у коді.
- `HomeView`: `.overlay { if isLoading { SplashView() } }` з плавним зникненням.

**Мінімальна тривалість (правка):** заставка зникала надто швидко при швидкому завантаженні.
Додано `splashMinimumDuration = 2.0`: у `loadData()` після фетчів обчислюється залишок часу
до 2 с і за потреби `Task.sleep`, після чого `isLoading = false`. Затримка діє лише коли
заставка реально показана (`showsSplash = isLoading` на вході). Помилка фетчу обробляється
власним `do/catch` (а не `defer`), бо для затримки потрібен `await`, якого `defer` не дозволяє.
Анімацію зроблено «продуманою»: поява лого (spring), обертове «дихаюче» кільце (rotation +
trim), пульсація, назва застосунку — щоб відпрацьовувала за ~2 с.

**Висновки:**
- Прапорець завантаження, що стає false лише один раз, уникає блимання заставки при
  поверненні на екран.
- Для мінімального часу показу splash рахувати `elapsed` і `Task.sleep` на залишок —
  `defer` тут не підходить (немає `await`); помилку ловити власним `do/catch`.
- Xcode 16 з `PBXFileSystemSynchronizedRootGroup`: нові файли й ассети в синхронізованій
  теці підхоплюються автоматично — правки `project.pbxproj` не потрібні.

## 2026-06-29 — Англійська локалізація + перемикач мови в застосунку

**Контекст:** уся UI була захардкоджена українською (~39 файлів). Додано повну
англійську локалізацію з перемикачем мови (УК/EN) у Налаштуваннях, без перезапуску.

**Архітектура:**
- `Localizable.xcstrings` (String Catalog), `sourceLanguage = uk`, переклади `en`.
  Ключі — самі українські рядки (мінімум правок у `Text("…")`, бо вони локалізуються
  автоматично як `LocalizedStringKey`).
- `developmentRegion = uk`, `knownRegions += uk` у `project.pbxproj`.
- Runtime-перемикання — підміна класу `Bundle.main` на підклас, що форвардить
  `localizedString(forKey:…)` у вибрану `.lproj` (`Bundle.setLanguage`). Root-view
  оновлюється через `.environment(\.locale,)` + `.id(language)`.

**Ключовий висновок (split-brain!):** `Text("літерал")` проходить через swizzle
`Bundle.localizedString` і поважає внутрішній перемикач, але **`String(localized:)`
його ІГНОРУЄ** — резолвиться через системні `AppleLanguages`. Тому всі `String(localized:)`
у ViewModel/моделях замінено на власний хелпер `L(_:)`, що викликає саме
`Bundle.main.localizedString(...)`. Без цього частина екрана англійською, частина — ні.

**Інші нюанси:**
- `Text(змінна)` / хелпери з `title: String` НЕ локалізуються — або змінити параметр на
  `LocalizedStringKey`, або обгорнути `Text(LocalizedStringKey(value))`, або `L(value)`.
- Форматери: жорсткий `Locale(identifier: "uk_UA")` → `LocalizationManager.shared.locale`
  (валюта ₴ лишилась). CSV-експорт лишено `uk_UA` — `yyyy-MM-dd` локале-незалежний.
- Сід (`Category.defaultCategories`, дефолтні рахунки) локалізується за мовою на момент
  першого запуску через `L(name)`. Вже створені (стара БД / CloudKit) дані не
  переперекладаються — лишаються мовою, якою були засіяні.
- Інтерпольований `Text("…\(str) · \(int)…")` генерує ключ із `%@`/`%lld` (непозиційні) —
  ключ у каталозі має точно збігатися, інакше переклад не підхопиться.
- Віджет (`WidgetShared`) — окремий таргет без каталогу; лишено українською (іде за
  системною мовою).

## 2026-06-30 — Source-мова не локалізується при іншій системній мові

**Контекст:** при обраній у застосунку українській (source-мова) екрани показувалися
англійською, якщо глобальна мова телефону — англійська. Інші мови працювали.

**Причина:** для source-мови `uk` не генерується `uk.lproj` (ключі вже українські).
`Bundle.setLanguage("uk")` лишав порожній associated bundle, а `LocalizedBundle`
падав у `super.localizedString(...)`, який резолвить рядок за **системною** мовою
телефону → знаходив українські ключі в `en.lproj` і повертав англійські значення.

**Висновок:** при підміні `Bundle.main` для source-мови не можна делегувати в `super` —
треба повертати сам ключ (`value` якщо непорожній, інакше `key`), імітуючи семантику
«переклад не знайдено» без системного fallback. Виправлено в обох копіях
`Bundle+Language.swift` (FlowCash + FlowCashW).
